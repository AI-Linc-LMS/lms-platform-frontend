# Assessment-Management FRONTEND — Revamp RCA & Phased Plan

> Full-module FE redesign to the adaptive-course design language + loophole RCA. Generated by a 10-agent workflow (8 surface audits). Design-extraction agent stubbed; design targets confirmed from the synthesis (real adaptive primitives).

## Executive summary

The assessment-management frontend is a functionally rich but structurally aging admin module: seven surfaces (list, 3-step create/edit wizard, edit tabs, submission-detail manual eval, live-monitor, plus modals) built as large bespoke MUI monoliths (create/edit pages are ~2050 and ~2783 lines). Two theming systems fight throughout — CSS custom properties (var(--accent-indigo), color-mix) and raw MUI palette (primary.main, alpha()) — with repeated inline sx, duplicated components (two paginations, two SectionCards, unused DeleteConfirmationModal), and per-file re-implementations of the same table/chip/empty-state. On top of the visual drift sit real defects: several are high-severity data-loss (composed email silently dropped on Create; unsaved evaluation marks discarded on Publish; draft email/reminders lost on reopen; AI-generation clobbering live edits) plus correctness bugs (Rules-of-Hooks crash on deleting the last section, stale-closure Next gating, select-all across pages) and security gaps (CSV formula injection, missing UTF-8 BOM, unsanitized dangerouslySetInnerHTML). The adaptive-course design language already ships as reusable primitives (AdaptiveSectionShell/Hero, SectionHero/KpiRail, Reveal/CountUp, common/Card, ConfirmDialog, PageShimmer). Strategy: (1) ship a no-visual correctness hotfix first to stop data-loss and crashes; (2) build a small set of assessment-scoped shared primitives once (adopting the adaptive-course tokens/components); (3) convert each surface into a thin consumer, phase by phase, each phase independently deployable to staging, folding that surface's remaining loopholes into the same PR.

## Shared foundations (build once, reuse everywhere)

- AssessmentPageHeader — one header component wrapping AdaptiveSectionHero (chapter eyebrow + gradient icon badge + title + subtitle + rightSlot for actions + breadcrumb/back). Replaces the bespoke gradient-clip h4 on list, the inline back-button+h4 on create/edit, and the three divergent hero treatments on the submissions/analytics/detail surfaces.
- AssessmentShell — thin wrapper over AdaptiveSectionShell + AdaptiveCardBackdrop (radius-4 card, hairline border, radial mesh, backdrop blur) so every surface sits on the same backdrop; MainLayout fullWidthContent, maxWidth ~1760.
- SectionCard / SurfaceCard — single tokenized card primitive over components/common/Card.tsx (header slot + icon tile + body), with a type-accent variant prop (indigo/success/warning) replacing every hand-rolled Paper+gradient+color-mix header across BasicInfoSection, AssessmentSettingsSection, MultipleSectionsSection, sidenav, preview.
- AssessmentDataTable — one data-table primitive (tokenized header, zebra/hover, sticky header, labeled action column, contained horizontal scroll, responsive column-priority/stack, skeleton rows) replacing the ~7 separate MUI Table implementations across list, create-preview, edit Questions, submissions, and analytics.
- StatusChip / Badge set — canonical semantic chip components (StatusChip, DifficultyChip[Easy=success/Medium=warning/Hard=error], SourceChip, UsageChip, OptionChip, CountBadge) with one status→token map; deletes all inline color-mix chip literals and the duplicated chip-color helpers, and fixes the Hard=Medium color collision and the #0d9488 hard-coded hex.
- SegmentedTabs — adaptive segmented/pill tab control with a single URL-driven source of truth, replacing MUI underline Tabs and the ToggleButtonGroup (with its !important overrides) on edit tabs, question sub-tabs, mode switcher, and the section-kind switcher.
- WizardStepper — token-driven step indicator with active/complete/error states (and click-to-jump for completed steps), replacing the heavily sx-overridden MUI Stepper in the create/edit wizard.
- EmptyState — one component (icon/illustration + title + description + primary CTA) replacing the ~10 divergent empty states (dashed Paper, warning-tinted 'no questions', bare Typography), consistent across breakpoints.
- LoadingSkeleton / adopt PageShimmerLayout — table/filter/form-shaped skeletons reused in-page (not just route loading.tsx) to kill the spinner→content layout jump on list, create draft-load, edit, submissions, and live-monitor.
- AssessmentFilterBar / Toolbar — tokenized filter/toolbar (consistent control sizing, localized labels, active-filter chips) for the list filter bar and the question-bank FacetBar.
- Consolidated ConfirmDialog — standardize on components/common/ConfirmDialog.tsx for delete/duplicate/clear-all/revoke; delete the unused DeleteConfirmationModal.tsx, the duplicate PaginationControls/AssessmentPagination pair, native confirm() calls, and the legacy QuizSectionSection/second SectionCard.
- Shared Pagination — one pagination component (Showing X–Y of Z + PerPageSelect) replacing the three inconsistent implementations (MUI Pagination, TablePagination, bespoke footer).
- Form primitives — a single Field / FieldGroup / helper-text set imported once (currently duplicated verbatim in BasicInfoSection and AssessmentSettingsSection).
- csvExport util — one hardened helper: prepends UTF-8 BOM and neutralizes formula injection (leading = + - @ tab/CR) for every export (list submissions+questions, edit submissions, analytics).
- useUnsavedChangesGuard hook — dirty-flag tracking + in-app navigation confirm + beforeunload, reused by create wizard, edit page tab-switch/back, and submission-detail evaluation.
- Button variants + KpiRail/Reveal adoption — adaptive primary/secondary/ghost/destructive pill buttons (no per-instance sx) and reuse of KpiRail/CountUp/Reveal for stat rows and entrance motion.
- Async-data helpers — a useServerPage hook (server-side page/limit/search/status params) for the list and question banks, plus a standard inline error+retry render used by every async source.

## Phased plan

### Phase 0 — Critical correctness & data-loss hotfix (no visual change)
**Goal:** Stop active data-loss, a crash, and security exposure immediately, decoupled from the visual redesign so it can ship the same day to staging and prod.
**Surfaces:** Create/edit wizard (email pipeline, Next gating, draft reload), Step 2 sections & question modes, Submission-detail manual evaluation, Assessment list page exports & pagination, Edit page publish gating
- Fix the email pipeline: lift EmailNotificationEditor subject/body/attachment/reminders to page-level state (or keep the editor mounted across steps) so Create/Publish always snapshot real content and email_notification_enabled can't disagree with the payload; rehydrate draft email + reminders + unresolved course ids on reopen.
- Move the SectionBasedQuestionsInput sections.length===0 early-return below all hooks to fix the Rules-of-Hooks crash on deleting the last section.
- Guard the manual-evaluation Publish (save-before-publish or disable-while-dirty) and add an unsaved-changes guard to the eval page.
- Add formula-injection escaping + UTF-8 BOM to every CSV export via a shared csvExport util; sanitize the email-trigger HTML preview.
- Fix select-all across pages, editingIndex desync (stable ids), AI-generation clobbering (disable edits while generating + unmount guard), and coding-CSV incremental commit.
- Fix list-page email-trigger send to use buildEmailBody, clamp list page after delete/duplicate, and add the missing tabSwitchLimit deps to isNextButtonDisabled.
- Gate Publish/Publish-evaluated/Evaluate on !readOnly; only allow publishing submitted/evaluated submissions with a confirm + in-flight guard.

### Phase 1 — Shared foundations + Assessment list page
**Goal:** Build the assessment-scoped design-system primitives once (adopting adaptive-course tokens/components) and prove them by converting the simplest surface — the list page — into a thin consumer.
**Surfaces:** Shared primitives library (components/admin/assessment/shared), Assessment list page (/admin/assessment)
- Build AssessmentPageHeader, AssessmentShell, SectionCard, AssessmentDataTable, StatusChip/DifficultyChip/CountBadge, SegmentedTabs, EmptyState, LoadingSkeleton (adopt PageShimmerLayout), AssessmentFilterBar, shared Pagination, consolidated ConfirmDialog, Field/FieldGroup, Button variants, csvExport util, useUnsavedChangesGuard, useServerPage — grounded on AdaptiveSectionShell/Hero, SectionHero/KpiRail, common/Card, common/ConfirmDialog.
- Rebuild the list page on these primitives: PageHeader replaces the gradient-clip h4; FilterBar replaces the bespoke flex of Select/Chip; one DataTable+row/card definition and one shared action menu replace the duplicated desktop-table/mobile-card+menu; in-page skeleton replaces the bare spinner; distinct inline error+Retry state separate from EmptyState (with a Create CTA).
- Delete dead/duplicate code surfaced here: DeleteConfirmationModal.tsx, the PaginationControls/AssessmentPagination pair; centralize the create-permission role check; localize labels and date formatting.
- Fix remaining list loopholes: email-job map recency (prefer latest job) and stop clearing the map on fetch error; guard empty-export downloads; aria-label the mobile overflow menu; rename canTrigggerEmailJob; tokenize the #0d9488 hex and color-mix literals; move filter/pagination server-side (or debounce).

- Live-monitor: wrap in AssessmentShell + AdaptiveSectionHero (eyebrow, gradient icon badge, subtitle, rightSlot), add KpiRail (Connected/Filtered/Focused/Auto-refresh) + Reveal entrance, gradient pill buttons, adaptive video-tile cards; make tiles keyboard-accessible (ButtonBase/role/onKeyDown/aria-label); fix pagination clamp (setPage(totalPages) not 1); scope 'Mute all audio' to focus view and collapse audio state to one boolean; handle focused-student disconnect (banner + Back-to-grid); add token-refresh/Reconnect for expired subscriber tokens; memoize participant display-name parsing; PageShimmer for loading/connecting.

## Per-surface audits

### Assessment list page (/admin/assessment) — table, filters, create entry, delete/duplicate/email-trigger/export
_Files:_ app/admin/assessment/page.tsx, app/admin/assessment/loading.tsx, components/admin/assessment/AssessmentTable.tsx, components/admin/assessment/AssessmentPagination.tsx, components/admin/assessment/PaginationControls.tsx, components/admin/assessment/DeleteConfirmationModal.tsx

The page is a client component (`"use client"`) wrapped in MainLayout(fullWidthContent). Layout top-to-bottom: (1) a header row with a gradient-clipped h4 title + subtitle on the left and a contained "Create Assessment" button on the right (disabled for course/content managers); (2) a Paper "filter bar" containing a search TextField plus five MUI Select dropdowns (Status, Authoring/draft, Proctoring, Payment, Evaluation), a conditional "Clear" button, a row of removable filter Chips, and a "Showing N of M" count that only appears when filters are active; (3) either a centered CircularProgress (while loading) or a Paper wrapping AssessmentTable + AssessmentPagination.

Data flow: on mount it fires two independent effects — loadAssessments() (fetches ALL assessments for the client in one call) and loadAssessmentEmailJobs() (fetches all email jobs, reduced to a map keyed by assessment_id). Filtering is done client-side in a useMemo over the full array; pagination is also client-side (slice by page/limit). A separate effect resets page to 1 whenever any filter changes.

AssessmentTable renders two entirely separate UIs behind useMediaQuery: a desktop MUI Table (Title/Courses/Duration/Questions/Status/Created/Actions with responsive column hiding) and a mobile stacked-card view. Each row shows the title, status/draft/paid/proctored/live chips, submission count, duration, question count, courses (truncated w/ tooltip), created date, start/end time badges, a copy-share-link IconButton, and a "…" overflow Menu (View/Edit, Live monitor, Certificates, Manage re-attempts, Trigger/View/Retry Email Job, Download Questions, Download Submissions, Duplicate, Delete). Row actions drive four dialogs on the page: an inline Delete confirmation Dialog, a Duplicate confirmation Dialog, and a large Trigger-Email Dialog with an EmailTemplatePreview. Exports build CSVs client-side and trigger browser downloads. There is a route-level loading.tsx that renders a PageShimmerLayout list skeleton, but the in-page loading state uses a bare spinner instead.

**Loopholes:**
- [high/bug] WYSIWYG violation on email trigger: the confirmation dialog PREVIEWS buildEmailBody(assessment) (reads assessment.email_body or a fallback HTML template), but on confirm it SENDS email_body: (assessmentToTriggerEmail as any).email_html — a different, un-typed field that is very likely undefined. The admin approves one body and the backend receives another (possibly empty/undefined) one.  
  `@ app/admin/assessment/page.tsx:549 (send) vs page.tsx:1190 / buildEmailBody 468-520 (preview)` → Send the exact value that was previewed: pass email_body: buildEmailBody(assessmentToTriggerEmail) (with the same {name} handling the backend expects). Remove the cast to email_html or reconcile which field is canonical, and add a typed field on the Assessment interface.
- [high/state] Page is not clamped after deletions. Deleting/duplicating does not touch filters, so the reset-page effect does not fire. If an admin deletes the last row on page 3 and the list now spans 2 pages, page stays 3: paginatedAssessments slices an empty window (table shows 'No assessments found') while MUI Pagination gets count=2 page=3 (out-of-range, console warning, no selected page). The admin sees an empty table with data still present.  
  `@ app/admin/assessment/page.tsx:680-684 (paginatedAssessments), 587/621 (loadAssessments after mutate)` → Clamp page to totalPages in a useMemo/effect: const totalPages = Math.max(1, Math.ceil(filteredAssessments.length/limit)); if (page>totalPages) setPage(totalPages). Or after successful delete/duplicate, recompute and setPage(Math.min(page, newTotalPages)).
- [high/perf] Unbounded client-side list + pagination. getAssessments(config.clientId) fetches the ENTIRE assessment set in one request with no limit/offset; filtering and pagination are done in-memory (useMemo slice). With hundreds/thousands of assessments this is a large payload, and every keystroke in search re-filters the whole array and re-renders the table. loadAssessmentEmailJobs similarly fetches all jobs.  
  `@ app/admin/assessment/page.tsx:115-125, 630-684` → Move to server-side pagination/filtering (pass page/limit/search/status params to the API) or, if the dataset is genuinely small, cap it and virtualize the table; at minimum debounce the search input before filtering.
- [high/data-loss] Submissions CSV export lacks a UTF-8 BOM (uses a raw Blob at page.tsx:275), unlike the questions export which prepends '﻿' via downloadCsv. Excel on Windows will misdecode non-ASCII bytes, corrupting student names, and any ₹ / accented characters in exported submission data.  
  `@ app/admin/assessment/page.tsx:274-281` → Route the submissions CSV through the same downloadCsv helper (which adds the BOM), or prepend const BOM='﻿' to the blob content.
- [high/validation] CSV formula injection: escapeCsv only quotes on comma/quote/newline; it does not neutralize leading =, +, -, @, or tab. A student name/email/answer beginning with '=' or '+' becomes an executable formula when the exported CSV is opened in Excel/Sheets (classic CSV-injection / data-exfil vector).  
  `@ app/admin/assessment/page.tsx:128-134 (escapeCsv), used by both exports` → In escapeCsv, if the stringified value starts with one of = + - @ (or tab/CR), prefix it with a single quote or a tab and then quote it, so spreadsheets treat it as text.
- [medium/state] assessmentEmailJobMap keeps only the LAST job seen per assessment_id while iterating the jobs array (map[aid] = ...), with no ordering by recency. If an assessment has multiple jobs, the map can retain a stale or failed status depending on array order, driving the wrong menu item (Retry vs View vs Trigger).  
  `@ app/admin/assessment/page.tsx:97-113` → Sort jobs by created_at/started desc before reducing, or explicitly prefer the most recent job per assessment_id, so the row reflects the latest job state.
